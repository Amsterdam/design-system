# Responsive design

Components can adapt their layout or appearance to the available space.

We use two techniques for this:

- A media query applies styles when the browser window (the viewport) reaches a certain width; these widths are called breakpoints.
- A container query does the same, but measures the width of a parent element instead of the window.

## Adapt to the width of the window

Components can change their layout at two viewport widths.

### Our two breakpoints

Variables for these breakpoints are available in Sass.
Not as CSS custom properties, since those cannot be used in media queries.
They are defined in ‘rem’ units to respect changes that users may make to their default font size.

- `$ams-breakpoint-medium: 37.5rem;`
- `$ams-breakpoint-wide: 72.5rem;`

They compute to 600 and 1160 pixels if the root font size is 16.
The values are based on steps of 280 pixels, starting with a base of 40 pixels.
So, 600 = 40 + 2 × 280 and 1160 = 40 + 4 × 280.

To complement these, the [Page](/docs/components-containers-page--docs) container component sets a maximum width of `90rem`, which is 1440 pixels = 40 + 5 × 280.
For internal websites, this is `120rem` or 1920 pixels.

### Usage across the design system

Our design system uses a mobile-first approach.
Components are styled for the smallest viewports and adapt fluidly to use more space if available.

Most importantly, our [Grid](/docs/components-layout-grid--docs) uses these breakpoints to change its number of columns from 4 to 8 and then to 12.
This already goes a long way to make our pages usable on all kinds of devices.
It keeps the widths of the columns within a range and allows positioning content below or next to each other.

Some other components still use the breakpoints themselves to adapt their appearance further, like
[Breakout](/docs/components-layout-breakout--docs),
[Dialog](/docs/components-containers-dialog--docs),
[Image Slider](/docs/components-media-image-slider--docs),
[Menu](/docs/components-navigation-menu--docs),
[Page Header](/docs/components-containers-page-header--docs) and
[Page Footer](/docs/components-containers-page-footer--docs),
[Progress List](/docs/components-containers-progress-list--docs), and
[Tab Navigation](/docs/components-navigation-tab-navigation--docs).

### Use them in your own components

Reuse our breakpoints in your custom components, so that layout changes only occur on these two window widths.

We leave out the [media type](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/At-rules/@media#media_types) so that our media queries work for both screens and paged media.
This helps printed and PDF outputs look more like the web page.

#### With Sass

Import the variables from [src/common/breakpoint.scss](https://github.com/Amsterdam/design-system/blob/main/packages/css/src/common/breakpoint.scss) in the CSS package.
Use `min-width` for the media query.

```css
.my-component {
  /* General declarations and those for narrow viewports */

  @media (min-width: $ams-breakpoint-medium) {
    /* Overrides for medium viewports */
  }

  @media (min-width: $ams-breakpoint-wide) {
    /* Overrides for wide viewports */
  }
}
```

#### Without Sass

You’ll have to copy the values into your codebase and update all instances if we change them.

```css
.my-component {
  /* General declarations and those for narrow viewports */

  @media (min-width: 37.5rem) {
    /* Overrides for medium viewports */
  }

  @media (min-width: 72.5rem) {
    /* Overrides for wide viewports */
  }
}
```

## Adapt to the width of a container

Container queries let a component adapt to the width of its container rather than the viewport.
This is useful when a component is placed in different layout contexts – for example, inside a narrow dialog versus a wide page column.

### The utility class

We provide a utility class to establish an [inline-size query container](/docs/utilities-css-query-container--docs).

```css
.ams-query-container-inline-size
```

Apply it to any element to register it as a named container that descendant components can query.

### Usage across the design system

Several of our core container components already establish an inline-size query container:

- [Page](/docs/components-containers-page--docs) – so content inside can respond to the page’s available width independently of the viewport.
- [Dialog](/docs/components-containers-dialog--docs) – so components inside the dialog can adapt to its width, which varies from the viewport.
- [Grid Cell](/docs/components-layout-grid--docs) – so content inside a cell can respond to its column width rather than the full grid width.

All three share the same container name (`ams-query-container-inline-size`), so a single `@container` rule applies consistently no matter which of them is the nearest ancestor.

Components that live inside a query container can adapt their own layout too.
[Description List](/docs/components-text-description-list--docs) switches between a stacked and a side-by-side layout based on the width of its container.

### Use it in your own components

Write `@container` rules in your CSS to style elements based on their container’s inline size.
Choose the width that best fits the component – container queries don’t need to use our two breakpoint values.
Use `min-inline-size` and the range syntax for the media query.

```css
.my-component {
  /* Styles for narrow containers */

  @container ams-query-container-inline-size (inline-size >= 32rem) {
    /* Overrides for wider containers */
  }
}
```

If your component can appear both inside and outside one of these named containers, omit the container name to query the nearest ancestor of any kind.

```css
.my-component {
  @container (inline-size >= 37.5rem) {
    /* Styles when any ancestor container is at least 600px wide */
  }
}
```
